## 1. 产品概述

### 1.1 产品定位

MatuX 桌面端是面向 **学生** 的 AI 编程与 STEM 学习客户端，基于 Electron 框架封装 Angular Web 应用，内嵌 Python 后端服务，为大屏场景（PC/笔记本）提供沉浸式学习体验。

> **角色范围**：MatuX 仅有 **学生 (student)** 一个角色。机构管理功能已解耦至独立项目 `OpenMTEduInst`。

### 1.2 与移动端的关系

| 维度 | 桌面端 (Electron) | 移动端 (Flutter) |
|------|-------------------|------------------|
| 定位 | 编程 + STEM 实验主力平台 | 课程学习 + 日常练习 |
| 屏幕 | 1920×1080 大屏 | 手机/平板触屏 |
| 核心场景 | 代码编写、3D 实验、创作 | 视频课程、测验、进度查看 |
| 输入方式 | 键盘 + 鼠标 | 触屏 |
| 离线能力 | 全功能离线（内嵌后端） | 缓存模式离线 |

### 1.3 核心价值

| 维度 | 价值描述 |
|------|----------|
| **IDE 级编程体验** | Monaco Editor + AI 代码补全，键盘快捷键驱动 |
| **沉浸式实验** | GPU 加速的 AR/VR 实验室、数字孪生、电路仿真 |
| **离线可用** | 内嵌 Python 后端 + PWA 缓存，断网也能写代码做实验 |
| **多窗口创作** | 代码编辑、3D 预览、课程视频可拆分窗口 |
| **成果导出** | 项目文件本地保存、作品分享、学习报告生成 |

### 1.4 目标用户

**唯一角色：学生 (student)**

- 年龄范围：8-18 岁（少儿编程 + 青少年 STEM）
- 设备：Windows/macOS 笔记本或台式机
- 场景：在家自学、学校机房、编程兴趣班

## 2. 用户画像与场景

### 2.1 用户画像

| 画像 | 描述 | 核心诉求 |
|------|------|----------|
| **编程初学者** | 8-12 岁，零基础 | 可视化编程（Blockly）、趣味引导、即时反馈 |
| **编程进阶者** | 12-16 岁，有基础 | 代码编辑器、AI 辅助、项目实战 |
| **STEM 爱好者** | 10-18 岁 | 电路实验、AR/VR 探索、数字孪生 |

### 2.2 典型使用场景

#### 场景 A：编程入门 — Blockly 可视化编程
1. 小明（10 岁）打开 MatuX 桌面端，登录账号
2. 进入「创意引擎」模块，选择 Blockly 积木式编程
3. 拖拽积木块完成"让角色走到终点"的任务
4. 点击运行，看到 3D 角色按指令移动
5. 系统自动记录进度，获得成就徽章

#### 场景 B：代码进阶 — Monaco Editor + AI 辅助
1. 小红（14 岁）打开 MatuX，进入 AI 教育模块
2. 在代码编辑器中编写 Python 代码解决算法题
3. AI 代码补全实时提示，遇到错误时 AI 给出修复建议
4. 防作弊系统确保独立完成
5. 完成后查看学习报告，看到自己的进步曲线

#### 场景 C：STEM 实验 — 电路仿真
1. 小刚（15 岁）进入「电路虚拟实验」模块
2. 从元件库拖拽电阻、LED、电源到面包板
3. 连接电路，系统实时仿真电流和电压
4. 完成后获得积分，解锁更复杂的实验

#### 场景 D：沉浸式探索 — AR/VR 实验室
1. 小美（13 岁）进入「数字孪生实验室」
2. 加载 3D 太阳系模型，用鼠标旋转、缩放观察行星
3. 切换到 AR 模式（需摄像头），将行星投射到桌面上
4. 完成探索任务，获得探索者成就

#### 场景 E：离线学习
1. 小华在无网络环境下打开 MatuX
2. 浏览已缓存的课程内容，继续上次的编程练习
3. 本地代码编辑器正常工作，项目保存到本地
4. 恢复网络后，学习进度自动同步到云端

### 3.3 P1 功能（第二阶段）

#### F-08-AI：AI 个性化教师（P0 核心新增）

> **核心理念**：AI 教师不是一次性问答机器人，而是伴随学生整个学习旅程的**智能导师**。它了解每个学生的学习画像、记住每次对话、追踪成长轨迹，并提供个性化教学建议。

##### 3.3-AI.1 学生学习画像（Student Learning Profile）

**目标**：自动采集学生在平台上的所有学习行为，构建多维度的学习画像，让 AI 教师"认识"每个学生。

| 维度 | 数据来源 | 采集方式 | 更新频率 |
|------|----------|----------|----------|
| **年级/年龄** | 注册信息 | 手动填写 | 一次性 |
| **学习风格** | 行为分析 | VARK 模型推断（视觉/听觉/读写/动觉） | 每周更新 |
| **编程能力** | 代码提交记录 | 语法正确率、完成速度、复杂度 | 实时 |
| **知识掌握度** | 测验成绩 + 课程进度 | 每个知识点的掌握百分比 | 每次测验后 |
| **薄弱环节** | 错误日志分析 | 高频错误类型、反复错的知识点 | 每日 |
| **学习偏好** | 交互行为 | 偏爱视频/文字/互动/项目式 | 每周 |
| **兴趣倾向** | 课程选择 + 浏览行为 | TF-IDF 兴趣向量 | 每日 |
| **学习节奏** | 学习时段 + 时长 | 活跃时间窗口、单次持续时长 | 实时 |
| **注意力特征** | 答题行为 | 切屏频率、答题犹豫时间 | 每次测验后 |
| **情感状态** | 对话 + 交互 | 情绪倾向分析（积极/沮丧/困惑） | 每次对话 |

##### 3.3-AI.2 上下文文档保存（Context Document Store）

**目标**：为每个学生保存长期学习上下文，AI 教师能够"记住"学生的历史学习状态、过往问题和偏好，实现跨会话的连续教学体验。

**存储架构**：

```
Student Context Store
├── LongTermMemory (长期记忆 - 向量数据库)
│   ├── 学习摘要：每完成一个课程/模块，自动生成摘要
│   ├── 关键对话：重要问答对（如"终于理解了循环"）
│   ├── 里程碑事件：首次通过测验、完成第一个项目等
│   └── 能力评估快照：每月一次的能力快照
│
├── SessionMemory (会话记忆 - 内存)
│   ├── 当前对话上下文（最近 20 轮）
│   ├── 当前学习任务状态
│   └── 即时反馈状态
│
├── KnowledgeState (知识状态 - 结构化)
│   ├── 知识点图谱位置
│   ├── 已掌握/学习中/未接触 分类
│   ├── 前置知识缺口
│   └── 推荐学习路径
│
└── ContextDocuments (上下文文档 - 文件系统/对象存储)
    ├── 学生自述（"我的学习目标"）
    ├── 代码作品集
    ├── 学习笔记
    └── 项目文件
```

**技术实现方案**：

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 向量存储 | ChromaDB / LanceDB（本地嵌入式） | 桌面端本地运行，无需外部服务 |
| 嵌入模型 | text2vec-base-chinese / BGE-small-zh | 中文文本向量化，本地推理 |
| 摘要生成 | LLM (GPT-3.5/DeepSeek) | 课程完成后自动生成摘要并向量化 |
| 记忆检索 | 混合检索（向量相似度 + 关键词 + 时间衰减） | 结合语义和时效性 |
| 上下文注入 | System Prompt 模板 | 每次对话自动注入学生画像 + 相关记忆 |

**上下文检索流程**：

```
用户提问 "我还是不太懂循环"
    │
    ├─ 1. Session Memory ──→ 获取最近 5 轮对话（"刚才在讲 for 循环"）
    │
    ├─ 2. Vector Search ──→ 检索相关长期记忆
    │    query: "循环 不理解 困难"
    │    results: [
    │      "3天前：学生完成循环入门测验，for/while 正确率 60%",
    │      "1周前：学生提问 '为什么 for 和 while 有区别？'",
    │      "2周前：Blockly 循环积木关卡完成 80%"
    │    ]
    │
    ├─ 3. Knowledge State ──→ 知识状态
    │    "循环": mastery=0.45, status="learning"
    │    "前置-条件判断": mastery=0.85, status="mastered"
    │
    ├─ 4. Student Profile ──→ 学习风格
    │    learning_style="visual" → 多用图表/动画解释
    │
    └─ 5. AI 教师回复 ──→
         "小明，我记得你上周刚学完条件判断，掌握得很好！循环其实很像条件判断的重复版。
          上次你 for 循环的正确率是 60%，主要是 range() 的参数理解有点混淆。
          让我用一个动画帮你理清思路… [显示循环执行流程图]
          
          另外，你在 Blockly 里的循环积木用得很好，要不要试试把 Blockly 的逻辑
          翻译成 Python 代码？"
```

**数据持久化**：

| 数据层 | 存储位置 | 备份策略 |
|--------|----------|----------|
| 向量嵌入 | 本地 ChromaDB 文件 | 云端同步（可选） |
| 结构化记忆 | SQLite（桌面端本地） | 每 6h 增量备份到云端 |
| 对话历史 | SQLite + JSON 文件 | 实时同步 |
| 代码作品 | 本地文件系统 | 云端 Git 仓库（可选） |

##### 3.3-AI.3 个性化知识库（Personalized Knowledge Base）

**目标**：AI 教师的知识库不再是硬编码的静态内容，而是根据学生的学习阶段**动态调整**内容深度、呈现方式和关联推荐。

**知识库分层结构**：

```
┌──────────────────────────────────────────────────────┐
│  Layer 1: 全局知识库（所有学生共享）                    │
│  ├── 编程概念（Python / JavaScript / Blockly）         │
│  ├── STEM 知识（电路、机械、AI 基础）                   │
│  ├── 学习方法论                                        │
│  └── 常见错误与解决方案                                │
├──────────────────────────────────────────────────────┤
│  Layer 2: 阶段知识库（按年级/难度分层）                  │
│  ├── L1 (G1-G3)：图形化编程、基础逻辑                  │
│  ├── L2 (G4-G6)：Blockly → Python 过渡                │
│  ├── L3 (G7-G9)：Python 进阶、数据结构                 │
│  └── L4 (G10-G12)：算法、AI 入门、项目实战             │
├──────────────────────────────────────────────────────┤

##### 3.3-AI.5 智能教学建议（Intelligent Teaching Suggestions）

**目标**：AI 教师主动发现学生的学习问题，给出诊断和针对性建议，而非被动等待提问。

**诊断类型**：

| 诊断类型 | 检测方式 | AI 教师行动 |
|----------|----------|------------|
| **前置知识缺失** | 知识点图谱回溯 | "你正在学函数，但变量作用域还没掌握。要不要先回顾一下？" |
| **概念混淆** | 相似概念错误对比 | "你好像把 list 和 tuple 搞混了，它们最大的区别是…" |
| **学习高原期** | 能力曲线停滞 | "最近进度有点慢，换个方式试试？我准备了一个小游戏…" |
| **偏科预警** | 各维度不均衡 | "你的代码能力很强，但电路实验好久没碰了哦！" |
| **注意力下降** | 答题犹豫 + 切屏增加 | "学累了吧？休息 5 分钟，回来我教你一个有趣的实验！" |
| **过度依赖 AI** | AI 请求频率过高 | "试试自己先思考 2 分钟，我相信你能做出来！" |

**主动推送机制**：

```
触发条件检测（后端定时任务 / 事件驱动）
    │
    ├─ 即时推送（学习过程中）
    │   ├─ 连续 3 次相同类型错误 → 侧边栏弹出提示
    │   └─ 长时间未操作 → 问候 + 建议
    │
    ├─ 每日推送（每日首次登录）
    │   ├─ "今日学习建议"卡片（Dashboard 首位）
    │   ├─ 昨日回顾 + 今日目标
    │   └─ 薄弱知识点提醒
    │
    ├─ 每周推送（每周一）
    │   ├─ 周学习报告（桌面端原生通知）
    │   ├─ 进步最大的领域
    │   └─ 下周学习计划建议
    │
    └─ 里程碑推送（实时）
        └─ 成就达成 → 系统托盘通知 + Dashboard 庆祝动画

##### 3.3-AI.6 情感陪伴（Emotional Companion）

**目标**：AI 教师不仅是知识传授者，更是学习伙伴，能感知学生情绪，提供情感支持，维持学习动力。

**情感感知**：

| 输入信号 | 检测方式 | 情绪推断 |
|----------|----------|----------|
| 对话语气 | NLP 情感分析 | "这个好难啊😫" → 沮丧 |
| 答题行为 | 犹豫时间、修改次数 | 频繁修改 → 焦虑/不确定 |
| 学习频率 | 登录间隔 | 突然减少 → 失去兴趣/遇到困难 |
| 错误态度 | 错误后的行为 | 立即重试 → 积极；放弃 → 沮丧 |
| 显式反馈 | 对话中的 emoji/语气词 | "终于做出来了！🎉" → 兴奋 |

**AI 教师情感回应策略**：

| 学生情绪 | AI 教师策略 | 示例回复 |
|----------|------------|----------|
| 沮丧/挫败 | 共情 + 分解任务 + 降低难度 | "编程有时候确实让人头大，连我写了几百万行代码也会有这种感觉。咱们先把这个大问题拆成 3 个小步骤？" |
| 困惑/迷茫 | 换一种解释方式 + 关联已知知识 | "换个角度——循环就像你每天早上起床的流程：刷牙→洗脸→吃早饭，每天都在重复，对吗？" |
| 兴奋/成就感 | 强化正面反馈 + 记录里程碑 | "太厉害了！🎉 这是你独立解决的第 5 个算法题，我已经帮你记在学习日记里了！" |
| 焦虑/紧张 | 安抚 + 提供确定性 | "测验只是帮你发现哪里还需要加强，不是评判你。你上次测验进步了 15%，这次肯定也没问题！" |
| 无聊/失去兴趣 | 切换学习方式 + 引入游戏化 | "老看代码是挺无聊的。要不要试试用 Python 控制这个虚拟机器人走迷宫？" |
| 自信/主动 | 提供挑战 + 拓展边界 | "既然基础这么扎实了，想不想挑战一个真实的项目？比如用 Python 做一个简单的天气预报程序？" |

### 5.1 性能指标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 冷启动时间 | < 5 秒（含后端启动） | 双击到登录页 |
| 热启动时间 | < 2 秒 | 后端已运行时 |
| 页面切换 | < 300ms | Angular 懒加载路由 |

| 3D 渲染帧率 | ≥ 30fps | three.js 场景 |
| 内存占用 | < 500MB（含后端） | 空闲状态 |
| 安装包大小 | < 200MB | NSIS 安装包 |

### 5.2 兼容性

| 平台 | 最低版本 | 架构 |
|------|----------|------|

| Linux | Ubuntu 20.04+ | x64 |

- **数据同步**：网络恢复后自动上传离线记录

- 切换语言无需重启

- 键盘全操作（Tab / Enter / Escape / 方向键）
- 高对比度模式
- 屏幕阅读器兼容（ARIA 标签）
- 符合 WCAG 2.1 AA 标准

### 6.7 动效规范（参考原型站 Framer Motion）

| 动效 | 实现方式 | 场景 |
|------|----------|------|
| 页面切换 | `opacity + y: 20 → 0`，duration 0.5s | 路由切换 |
| 卡片入场 | `opacity + scale: 0.95 → 1`，stagger 0.1s | 列表加载 |
| 卡片悬停 | `scale: 1.05, y: -4`，spring 弹性 | 可点击卡片 |
| 按钮点击 | `scale: 0.95 → 1`，active 状态 | 所有按钮 |
| 浮动按钮 | `scale: 0 → 1`，初始弹入 | AI 助手入口 |
| 通知弹出 | `y: 50 → 0, opacity: 0 → 1` | Toast 通知 |
| 模态框 | `scale: 0.9 → 1, opacity: 0 → 1` | 对话框 |
| 进度条 | `width 过渡` + 渐变色填充 | 学习进度 |

### 6.9 设计原则

1. **品牌一致性**：与 `edu-proto` 原型站保持相同的色彩、字体、圆角、阴影体系
2. **聚焦学习**：界面围绕学生学习流设计，去除一切管理元素
3. **沉浸式编码**：代码区采用暗色终端风格（`bg-slate-900`），支持全屏禅模式
4. **趣味性**：emoji 图标、渐变色卡片、成就动画、Framer Motion 动效
5. **桌面端适配**：利用大屏优势，多列网格布局（4列快捷入口、3列推荐课程、2列任务+统计）
6. **响应式**：窗口缩放自适应，最小 1024×768 布局不错乱
7. **暗色模式**：支持亮色/暗色主题切换（参考原型站 `prefers-color-scheme: dark`）

### 6.10 桌面端 vs 移动端差异

| 设计维度 | 移动端（原型站 Phone 模式） | 桌面端（本 PRD 目标） |
|----------|---------------------------|----------------------|
| 导航方式 | 底部 Tab Bar (4 个图标) | 顶部 Navbar（文字链接 + 搜索 + 头像） |
| 快捷入口 | 2×2 网格 (4 个) | 1×4 网格 (4 个，大卡片) |
| 课程列表 | 单列垂直滚动 | 3 列网格 |
| 任务列表 | 纵向列表 | 左任务 + 右统计双栏 |
| 代码编辑器 | 全屏独占 | 左题目 + 右代码分栏 |
| AI 助手 | 全屏模态框 | 右侧浮窗面板 |
| 底部状态 | 无 | 状态栏（在线状态 + 版本号） |

## 9. 风险与依赖

### 9.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Python 环境依赖复杂 | 安装包过大、启动失败 | 中 | PyInstaller 打包为独立可执行文件 |
| 代码执行沙箱安全 | 恶意代码风险 | 中 | Docker 隔离 / 资源限制（CPU/内存/时间） |
| 3D 渲染性能不足 | 低配机器 AR/VR 卡顿 | 中 | 自动降级（检测 GPU，低配降画质） |
| Electron 版本兼容 | 系统 API 不可用 | 低 | 锁定 Electron 28 LTS |
| **LLM 服务不稳定** | AI 教师回复延迟或不可用 | 中 | 本地缓存常见回答 + 降级为规则回复 + 离线队列 |
| **向量数据库性能** | ChromaDB 检索慢影响 AI 响应 | 低 | 限制检索范围 + 定期清理过期记忆 + 异步检索 |
| **学生隐私数据** | 学习画像数据泄露风险 | 高 | 画像数据仅本地存储 + 端到端加密 + 可选择性云端同步 |
| **AI 教师回答质量** | 不准确或不当回答影响学习 | 中 | 教育领域 Prompt 工程 + 内容审核层 + 学生反馈机制 |
